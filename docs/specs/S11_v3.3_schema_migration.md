# S11 v3.3 -- Stress Field schema migration (BREAKING)

> Status: **design locked** 2026-06-22 (HEAD `v3.2.2`). Implementation in flight.
> Parent spec: [`S11_stress_field.md`](S11_stress_field.md). See also v3.2.2 handoff
> `docs/HANDOFF_v3.2.2.md` ¬ß 5 (v3.3 šłĽŤĽłŚÄôťĀł U-07) for upstream rationale.

## Why

`StressField::governingMemberId / governingShellId` reported the **user-assigned
element ID** (`Member::id` / `ShellQuad::id`) when an element governs, and `0`
when none governed (in-class default). Because user IDs are signed ints whose
value space includes `0`, the "no governing element" sentinel **collides with
a legitimate element ID of 0**.

- `StressField.cpp:102 / :163` already used `-1` as the sentinel internally
  (`worstMemberId / worstShellId`) and assigned the value found, but the
  in-class default of `0` in `StressField.h:78-79` meant a default-constructed
  `StressField` (no computation run, or all elements inactive) reported "member
  0 governs" -- indistinguishable from a real model whose `members[0].id == 0`
  truly governs.
- The v3.1.0 dispatcher audit fix (C-07/C-08) aligned the JSON wire to `-1`,
  but only via the cpp's `worstMemberId = -1` initialiser; the header default
  was never touched, so default-constructed `StressField` packed as JSON would
  still emit `governingMemberId: 0`.
- The v3.2 BP layer (FrameCoreUE `FFrameStressField`) used `INDEX_NONE` (= -1)
  on the USTRUCT side, with a clamp comment in `FrameCoreUEStressFieldLibrary.cpp`
  noting the "carries through when engine reports 0". A v3.2.x cantilever
  fixture passed only because its member `id == 1`, not `0`.
- v3.2.2 audit logged this as **U-07** with the explicit note: "fix needs
  engine source change, violates ťźĶŚČá #1, defer to v3.3".

The fix is to stop using user IDs for sentinel detection and switch to the
**internal element index** (`size_t e` over `model.members`, narrowed to
`int`), which is always `>= 0` for an existing element and unambiguous against
`-1`. The user ID stays available on each per-element record (`MemberStressTrace::memberId`,
`ShellStressLayer::shellId`); only the *governing pointer* changes meaning.

## Scope (4 layers, BREAKING)

### Layer 1 -- Engine struct (`StressField.h`)

| Before (v3.2.x) | After (v3.3) |
|---|---|
| `int governingMemberId = 0;    // 0 if no member governs` | `int governingMemberIdx = -1; // -1 if no member governs (index into FrameModel::members)` |
| `int governingShellId  = 0;    // 0 if no shell governs`  | `int governingShellIdx  = -1; // -1 if no shell governs (index into FrameModel::shells)` |

The two sibling fields (`governingShellLayer`, `governingShellCorner`) are
unchanged in name/type; `governingShellCorner` already used `-1` for "centre".

### Layer 2 -- Engine impl (`StressField.cpp`)

Rename the local accumulators and switch the value written to the index, not
the ID:

```cpp
// Before
int worstMemberId = -1;
...
worstMemberId = mem.id;             // <-- user ID
...
fld.governingMemberId = worstMemberId;

// After
int worstMemberIdx = -1;
...
worstMemberIdx = static_cast<int>(e);   // <-- internal index
...
fld.governingMemberIdx = worstMemberIdx;
```

Same change for the shell sweep (`worstShellId -> worstShellIdx`, `sh.id -> (int)s`).

Per-element records (`MemberStressTrace::memberId`, `ShellStressLayer::shellId`)
**keep the user ID** so a renderer can still draw element labels without a
second lookup. Both `memberIdx` and `memberId` are already on the per-element
record (see `StressField.h:42-43`) -- nothing changes there.

### Layer 3 -- Dispatcher v2 JSON (`Dispatcher.cpp:957-958`)

| Before (v3.2.x) | After (v3.3) |
|---|---|
| `dest.emplace("governingMemberId",  Json(...));` | `dest.emplace("governingMemberIdx", Json(...));` |
| `dest.emplace("governingShellId",   Json(...));` | `dest.emplace("governingShellIdx",  Json(...));` |

The JSON key changes in lockstep with the engine field name. We deliberately
**rename the key rather than keep `...Id` with new semantics** -- a value-only
silent break is strictly worse than a key rename, because clients that read
`governingMemberId` and look up `model.members[Id == value].id` would silently
mis-resolve every governing pointer.

`kEngineVer "3.2.0" -> "3.3.0"`. Capability set is unchanged (still 23
capabilities incl. `inspect.stress_field`); only the response body of
`inspect.stress_field` differs.

**Scope-out**: `solve.linear` `summary.governingMember` (D/C screen, line 618)
is **not changed**. On that path, `governingMember == 0` means "no element
fails the D/C screen" (a true semantics-of-zero, not a sentinel collision),
since the D/C path returns the worst D/C ratio across all elements and a value
of 0 there means no element exceeded its allowable. The convention there
predates v3.3 and there is no upstream demand to change it.

### Layer 4 -- FrameCoreUE USTRUCT (`FrameCoreUETypes.h`)

| Before (v3.2.x) | After (v3.3) |
|---|---|
| `int32 GoverningMemberId = INDEX_NONE;` | `int32 GoverningMemberIdx = INDEX_NONE;` |
| `int32 GoverningShellId  = INDEX_NONE;` | `int32 GoverningShellIdx  = INDEX_NONE;` |

BP accessor library functions (`UFrameCoreStressFieldLibrary`) follow:

| Before | After |
|---|---|
| `static int32 GetGoverningMemberId(const FFrameStressField& Field);` | `static int32 GetGoverningMemberIdx(const FFrameStressField& Field);` |
| `static int32 GetGoverningShellId(const FFrameStressField& Field);`  | `static int32 GetGoverningShellIdx(const FFrameStressField& Field);`  |

The marshal in `FrameCoreUEStressFieldLibrary.cpp` drops the "clamps to -1"
work-around comment (no longer needed -- engine now always emits -1 or a valid
index).

## Migration guide (consumers)

> ‚ö† **Legacy keys are ABSENT, not present with new values.** The v3.3 dispatcher
> emits **only** `governingMemberIdx / governingShellIdx`. A pre-v3.3 client that
> reads `governingMemberId` with a default-zero fallback will not crash ‚ÄĒ it will
> silently report "no governing element" (the old sentinel 0) for every model,
> including ones where an element actually governs. The
> `v2_roundtrip.py` fixture asserts both the new-key presence and the
> legacy-key absence, but unfixed clients only catch this by pinning
> `FRAMECORE_EXPECTED_ENGINE_VER='3.3.0'` against the dispatcher hello-response.

### GH / Rhino client (C# SDK)

Old code reading the JSON wire as `governingMemberId` and doing
`model.MembersById[id]` lookup:

```csharp
// v3.2 (broken when member id == 0 governs):
int gid = body["stressField"]["governingMemberId"].AsInt();
var gov = model.Members.FirstOrDefault(m => m.Id == gid);

// v3.3 (correct):
int gix = body["stressField"]["governingMemberIdx"].AsInt();
var gov = (gix >= 0) ? model.Members[gix] : null;
int govId = gov?.Id ?? -1;  // if you still need the user ID for labelling
```

### BP designer (FrameCoreUE)

Old node `Get Governing Member Id` -> new node `Get Governing Member Idx`.
If the BP graph then looked up the member by ID, replace the lookup with a
direct array index into `Field.Members[Idx]` (no search needed; one O(1) op).

### standalone CLI

The CLI does not expose `governingMember*` (no `INSPECT STRESS` CLI token in
`frame_cli_core.cpp`), so no CLI client impact. If one is added in v3.3+ it
will use the new `Idx` convention from day one.

## Deliberate non-changes

- Per-element record fields (`MemberStressTrace::memberId/memberIdx`,
  `ShellStressLayer::shellId/shellIdx`) keep both ID and index. The
  ambiguity bug only affected the **governing pointer at the field level**;
  per-element records were never ambiguous (a record's presence in the list
  means the element exists, so `id == 0` there is unambiguously "the element
  whose user-assigned ID is 0").
- `globalMaxFiberSigma` / `globalMaxVonMises` keep their `>= 0, == 0 if
  nothing screenable` contract. Zero is unambiguous here (there is no
  "real fiber sigma of negative magnitude" by the compression-positive
  convention, and the global-max is the max of `>=0` per-sample values).
- `governingShellLayer` (`Top` / `Bot` enum) and `governingShellCorner`
  (`-1` for centre, `0..3` for corners) are unchanged in semantics.
  `Corner == -1` for centre was always unambiguous (no real corner index is -1).

## Honest boundaries (additions on top of S11 main spec)

- **Migration is one-way**. A v3.2.x client reading `governingMemberId` from
  a v3.3 server gets a JSON parse miss (key absent), which the SDK should
  detect as `null` / default value. There is no compatibility shim emitting
  both keys; the v3.3 server emits **only** the `Idx` key. Rationale: dual
  emission would let unfixed v3.2 clients keep running with the same silent
  ambiguity bug -- worse than a loud "key not found" error that surfaces the
  required migration.
- **Server version pinning enforces the break**. `FRAMECORE_EXPECTED_ENGINE_VER`
  env var in `run_gate.ps1` / `run_gpu_gate.ps1` / `release-gate.yml` bumps
  from `'3.2.0'` to `'3.3.0'`; any client running against a v3.3 dispatcher
  must update its expected version to load the new schema.
- **No deprecation period**. The schema is new since v3.1.0 (only 2 minor
  versions old), no production deployments are known, and the bug is in the
  wire protocol itself -- so a clean break is preferable to a multi-release
  shim.
