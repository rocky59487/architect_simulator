## 4. Risk-adjusted ordering rationale

User-locked risk profile is **Experimental** with sub-band: **Safe** for patch path (#1-#5), **Experimental** for spike path (#6-#8). Experimental normally puts risky units FIRST (fail-fast), but dependency on `v0.3.1` base prevents that for spikes. Therefore:

- **Patch units (low-risk) run first** by necessity (release-hardening gates the spike units).
- **Within patch round 1**, all 3 units are LOW risk and independent → maximize parallelism.
- **Spike units run AFTER `v0.3.1` ships** so a sandbox blow-up doesn't strand patch work.
- **UE5.8 spike + Scenario-u1 dispatched in parallel** (independent codepaths, no file overlap) — if UE5.8 blows up in the sandbox, Scenario MVP continues unaffected.
- **GATE-A (UE5.8) is sequential to its spike** — user reviews report before any UE5.8 main-branch decision.
- **Scenario units u1 → u2 → u3 are sequential** because each builds on prior.
- **GATE-B (Scenario playability) gates v0.4.0** — if not playable, skip RELEASE-v0.4.0 entirely.

Critical-path bottleneck = SPIKE-UE5.8-eval (3-5h work-budget; may need sub-dispatch if external build dominates 30 min wall-clock).

Parallelism opportunities = Round 1 (3-way parallel) and Round 4 (2-way parallel UE5.8 + Scenario-u1).
