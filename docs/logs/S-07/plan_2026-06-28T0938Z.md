## 3. Risk-adjusted ordering rationale

**Why split into 2 units (overriding scope §2 "1 cohesive unit" preference):**

The scope contract suggested 1 unit with fallback split if iteration > 2. Phase 1 chose
to **pre-emptively split** for three reasons:

1. **Mechanical stops:** Phase 1 § 5 mechanical stop = 50 tool-call cap per dispatch.
   Combined work (test class write + UE build + isolated PASS verify + PowerShell wrapper
   + log parser + 6-leg full gate run) is easily 50+ tool calls if subagent is thorough.
   Split keeps each unit comfortably under the cap.
2. **Multi-domain:** u1 is C++ / UE Automation framework expertise; u2 is PowerShell
   / log-parser DevOps. Different SUBAGENT_PREFIX injections + different adversarial
   review focuses. Single-unit dispatch would mix concerns awkwardly.
3. **Phase 3 review quality:** Reviewing 2 smaller dispatches separately finds bugs
   more reliably than reviewing 1 mega-dispatch (this is the standard "smaller PR =
   better review" principle applied at /work granularity).

**Acceptable downside:** 1 extra dispatch round (Phase 2 dispatch + Phase 3 review)
adds ~20-30 min vs single dispatch. Worth it for the above 3 benefits.

**If user prefers original 1-unit:** they can choose "Modify" at the Tier 3 plan-lock
gate; Phase 1 will collapse u1+u2 back into a single 4-5.5h dispatch (will likely hit
step cap; subagent prompt must explicitly tell them to compress tool usage).

**Risk profile:** Experimental. Per Phase 1 ORDERING_RULES "Risky/unknown units
FIRST (fail fast)", u1 (the risky LatentCommand work) is dispatched first — its
failure mode is loud (test won't compile or won't PASS) and we save u2's 1.5h
if u1 needs to escalate. u2 is mechanical PowerShell once u1 stabilizes.

**Critical path:** strictly serial. Total ~4-5.5h. No parallel speedup possible
because u2 needs u1's verified-passing test as the wrapped artifact.
